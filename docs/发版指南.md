# Storyloom 发版指南

> 最后更新：v1.1.0

## ⚠️ 重要提醒

**Tauri 产物自动签名与更新元数据**

Tauri 构建系统（`tauri build`）会自动处理代码签名和更新元数据（`.sig` 文件）。与旧版 Electron 不同，Tauri 使用 `tauri-plugin-updater`，其更新检查通过 GitHub Release 的 `.sig` 文件和 `latest.json`（或 updater endpoint）完成。

构建完成后，检查 `src-tauri/target/release/bundle/nsis/` 目录下的产物，确认 `.exe` 和同名 `.sig` 文件均已生成。

---

## 1. 版本号策略

- **单线 SemVer 递进**：`major.minor.patch`，每次只在一条主线上前进。
- **`identifier` 不可变更**：Tauri 配置中 `src-tauri/tauri.conf.json#identifier` 始终保持 `com.storyloom.app`。该字段是 Windows 安装/升级链路的应用标识，已部署用户的覆盖安装、注册表项、用户数据目录均与之绑定，迁移成本远超收益。
- 资产命名采用连字符版（`Storyloom_X.Y.Z_x64-setup.exe`），由 Tauri 构建系统自动生成。

## 2. 标准发版流程（10 步）

> 假设目标版本号 `X.Y.Z`，以下命令按顺序执行；任意一步失败必须先修复再继续。

### Step 1 — Bump version

同步修改两处版本号：

```bash
# package.json
npm version X.Y.Z --no-git-tag-version

# src-tauri/tauri.conf.json 和 src-tauri/Cargo.toml（通常由 Tauri CLI 自动同步）
```

并同步 lockfile：

```bash
npm install --package-lock-only
```

### Step 2 — 质量门三连

```bash
npm run typecheck
npm run test
npm run build
```

三条必须 exit 0。任何 TS 错、单测失败、构建失败一律就地修。

### Step 3 — Commit

```bash
git add -A
git commit -m "chore(vX.Y.Z): <一句话总结本版主要改动>"
```

### Step 4 — Tag

```bash
git tag -a vX.Y.Z -m "Storyloom vX.Y.Z — <亮点>"
```

### Step 5 — Push

```bash
git push origin master
git push origin vX.Y.Z
```

### Step 6 — 构建分发包

```bash
# 构建前端 + 后端 + sidecar
npm run build
npm run build:server
npm run build:sidecar

# Tauri 构建（Windows NSIS）
npm run tauri:build
```

产物输出目录：`src-tauri/target/release/bundle/nsis/`

产物清单：
- `Storyloom_X.Y.Z_x64-setup.exe` — NSIS 安装程序（约 ~50MB）
- `Storyloom_X.Y.Z_x64-setup.exe.sig` — 签名文件（updater 用）

> **注意**：Tauri 构建产物体积从 Electron 时代的 ~131MB 降至 ~27MB，因为 Tauri 使用系统 WebView2 而非捆绑 Chromium。

### Step 7 — 验证产物

```powershell
# 检查产物是否存在
Test-Path "src-tauri/target/release/bundle/nsis/Storyloom_X.Y.Z_x64-setup.exe"

# 验证文件大小（应约 25-30MB）
(Get-Item "src-tauri/target/release/bundle/nsis/Storyloom_X.Y.Z_x64-setup.exe").Length / 1MB
```

### Step 8 — 上传 GitHub Release

```bash
gh release create vX.Y.Z \
  "src-tauri/target/release/bundle/nsis/Storyloom_X.Y.Z_x64-setup.exe" \
  "src-tauri/target/release/bundle/nsis/Storyloom_X.Y.Z_x64-setup.exe.sig" \
  --title "Storyloom vX.Y.Z — <主题>" \
  --notes-file docs/更新日志-vX.Y.Z.md
```

两件套：`.exe` + `.sig`。`.sig` 是 `tauri-plugin-updater` 验证签名必需的。

> **v1.4.0+ 签名说明**：由于 Tauri signer CLI 在 Windows 上存在兼容性问题，现使用 Python `pynacl` 手动实现 minisign 格式签名。签名脚本见 `scripts/sign-release.py`。

### Step 9 — 验证 updater 配置

检查 `src-tauri/tauri.conf.json` 中 updater 配置是否正确指向本仓库：

```json
{
  "plugins": {
    "updater": {
      "pubkey": "...",
      "endpoints": ["https://github.com/YJLZSL/Storyloom/releases/latest/download/latest.json"]
    }
  }
}
```

> 注意：Tauri updater 使用 `latest.json` 作为元数据文件。构建完成后需要手动生成 `latest.json` 和 `latest.yml`（见 `scripts/gen-latest.js`）。若尚未配置 updater endpoint，请参考 Tauri 官方文档配置静态 JSON 或自建更新服务器。

### Step 10 — 更新 CHANGELOG

把本版条目加到 `更新日志.md` 顶部，提交：

```bash
git add 更新日志.md
git commit -m "docs(changelog): add vX.Y.Z entry"
git push origin master
```

## 3. 自动更新调试

### 客户端日志

Tauri 日志通过 `tauri-plugin-log` 写入应用日志目录。查看方式：

```powershell
# 打开日志目录（由 Tauri API 自动管理）
# 默认位置：%APPDATA%\com.storyloom.app\logs\app.log
Get-Content "$env:APPDATA\com.storyloom.app\logs\app.log" -Tail 200 -Wait
```

前端可通过 `src/lib/tauri-api.ts` 中的 `writeLog()` 自定义日志级别。

关键日志前缀：
- `[updater] checking-for-update`
- `[updater] update-available v...`
- `[updater] error`

### 手动复现升级

1. 把本地 `package.json#version` 和 `tauri.conf.json#version` 临时改成上一版（例：发布 `1.3.1` 后想测，就改成 `1.3.0`）。
2. `npm run tauri:build`，安装产出的安装包。
3. 启动应用；`UpdateNotifier` 组件会自动 `checkUpdate()`。
4. 抓包：用 Fiddler / Charles 看是否请求了 updater endpoint。

### 常见坑

- `tauri.conf.json#identifier` 必须与实际应用标识一致（当前 `com.storyloom.app`）。**不要修改此字段**。
- `pubkey` 必须与签名私钥配对。**v1.4.0 重新生成了签名密钥对**，老版本用户需要手动下载新版安装包，之后可正常自动更新。
- 生产模式下 `autoDownload = false`，下载需要渲染层显式调用 `downloadUpdate()`；这是 UI 设计决策，不要随手改。
- **Sidecar 打包**：`npm run build:sidecar` 使用 `pkg` 将 Node.js 后端打包为 `src-tauri/sidecars/storyloom-sidecar.exe`。确保该文件存在且正确注册在 `tauri.conf.json#bundle.externalBin` 中，否则 Rust 主进程无法启动后端。
- **Sidecar 打包**：`npm run build:sidecar` 使用 `pkg` 将 Node.js 后端打包为 `src-tauri/sidecars/storyloom-sidecar.exe`。确保该文件存在且正确注册在 `tauri.conf.json#bundle.externalBin` 中，否则 Rust 主进程无法启动后端。
