### 架构要点

- **Tauri Rust 主进程**：负责窗口管理、系统级 API（文件/端口/日志/对话框）、sidecar 生命周期管理
- **Sidecar 子进程**：Fastify 后端作为独立 Node.js 进程运行，通过 stdout JSON 协议与 Rust 主进程通信
- **WebView2 渲染进程**：前端 React 应用运行在系统 WebView2 中，通过 HTTP 与 sidecar 通信
- **安全模型**：CSP 策略 + Tauri Capabilities 机制，前端无法直接访问系统资源，必须通过 Rust Commands


```
    │ HTTP fetch (localhost:PORT)
    ▼

    ▼
```

## 9. 开发注意事项

### 9.1 必须使用 `Edit` 工具修改文件
- 用 `Read` 读取文件后，使用 `Edit` 的 `old_string`/`new_string` 精确替换
- 不要猜测 `old_string`，必须从 `Read` 输出中复制
- 对于大文件，使用 `line_offset` 和 `n_lines` 分页读取

### 9.2 不要依赖 `drizzle` 迁移目录
- 项目没有 `drizzle/` 目录，不使用 `drizzle-kit generate`
- 数据库迁移在 `server/db/index.ts` 中通过硬编码 DDL 执行
- 新增表/列必须同时修改：schema.ts → index.ts DDL → ensureSchemaCompatibility

### 9.3 构建时序

必须先 `npm run build`（前端），再 `npm run build:server`（后端），再 `npm run build:sidecar`（sidecar 可执行文件），最后 `npm run tauri:build`（Tauri 桌面应用）。

```bash
# 完整构建流水线
npm run build          # Vite 前端 → dist/
npm run build:server   # tsc 后端 → dist-server/
npm run build:sidecar  # pkg 打包 → src-tauri/sidecars/storyloom-sidecar
npm run tauri:build    # cargo 编译 + NSIS 打包 → src-tauri/target/release/bundle/
```

`build:sidecar` 使用 `pkg` 将 `dist-server/server/sidecar-entry.js` 打包为独立可执行文件，放置于 `src-tauri/sidecars/storyloom-sidecar.exe`。该文件必须在 `tauri:build` 前存在，因为 Tauri bundle 会将 `sidecars/` 目录中的文件打包进安装程序。

### 9.4 Tauri 配置要点

`src-tauri/tauri.conf.json` 关键字段：

- `identifier`：`com.storyloom.app`（应用全局标识，不可变更）
- `build.frontendDist`：`../dist`（Vite 构建输出目录）
- `build.devUrl`：`http://localhost:5173`（开发模式前端地址）
- `bundle.targets`：`["nsis"]`（Windows 安装包格式）
- `app.windows`：主窗口配置（尺寸、最小尺寸、背景色）
- `app.security.csp`：内容安全策略（限制资源加载来源）

### 9.5 测试
- 测试框架：Vitest
- 测试文件：`*.{test,spec}.{ts,tsx}`
- 运行：`npm test`
- 当前通过：193/193（22 个测试文件）
