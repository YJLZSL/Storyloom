# Storyloom 本地开发指南

> 最后更新：v1.4.0（美术架构升级）

## 1. 环境要求

| 工具 | 版本 | 用途 |
|---|---|---|
| Node.js | 20.x LTS 或更高 | 运行 Vite / Fastify / sidecar |
| npm | 随 Node.js 自带 | 包管理（仓库使用 `package-lock.json`） |
| Rust | 1.77.2+ | Tauri 2.x 构建必需（`rustc` + `cargo`） |
| Python | 3.10+ | 运行 `scripts/generate-storyloom-icons.py` |
| Pillow | 11+ | 同上，图标合成 |
| Windows 10/11 (x64) | — | 跑 `npm run tauri:build` 出安装包必需 |
| Git | 任意近期版 | 源码管理 |

### Windows 特有要求

Tauri 2.x 在 Windows 上编译需要 **Visual Studio Build Tools 2022**，必须包含 **"Desktop development with C++"** 工作负载。安装方式：

1. 下载 [Visual Studio Build Tools](https://visualstudio.microsoft.com/downloads/#build-tools-for-visual-studio-2022)
2. 运行安装程序，选择 **"Desktop development with C++"** 工作负载
3. 等待安装完成（约 3-8GB）

> 注意：旧版 Electron 时代需要 `windows-build-tools` 或 `better-sqlite3` 编译工具；Tauri 时代需要 VS BuildTools 用于 Rust 编译 + WebView2。

校验：

```bash
node -v        # v20.x
npm -v
rustc --version  # 1.77.2+
cargo --version
python --version
git --version
```

## 2. 本地起步

```bash
git clone https://github.com/YJLZSL/Storyloom.git
cd Storyloom
npm install
```

> Windows 上首次 `npm install` 会编译 `better-sqlite3`，需要 Visual Studio Build Tools，10–60 秒不等。

## 3. 开发模式

| 命令 | 起什么 | 用途 |
|---|---|---|
| `npm run dev` | Vite (5173) + Fastify (3001) | 浏览器调 UI，最快反馈 |
| `npm run tauri:dev` | Tauri 桌面壳 + Vite + Fastify sidecar | 调桌面端 / Rust 命令 / sidecar 通信 |
| `npm run dev:sidecar` | 仅 Fastify sidecar（`tsx watch`） | 仅调后端 API |
> 注意：旧版后端开发模式（`npm run dev:server`）已废弃，请使用 `npm run dev:sidecar` 作为后端开发命令。`npm run dev:server` 仅兼容旧版非 sidecar 架构。

```bash
# 仅前端 + 后端：浏览器访问 http://localhost:5173
npm run dev

# 完整 Tauri 桌面开发模式（推荐日常开发）
npm run tauri:dev

# 仅 sidecar 后端
npm run dev:sidecar

# 仅后端（旧版模式）
npm run dev:server
```

`npm run tauri:dev` 内部由 Tauri CLI 协调时序：先启动前端 dev server，再启动 sidecar，最后打开 WebView2 窗口。Rust 源码修改会自动热重载（`cargo watch` 行为）。

## 4. 目录约定

```
src-tauri/                Tauri 2.x Rust 主进程
  ├─ src/
  │   ├─ main.rs          入口（run()）
  │   └─ lib.rs           命令注册 + sidecar 启动 + 日志
  ├─ Cargo.toml           Rust 依赖（tauri + plugins + tokio）
  ├─ tauri.conf.json      Tauri 配置（窗口 / bundle / updater）
  ├─ icons/               应用图标（多尺寸 PNG + ICO + ICNS）
  └─ sidecars/            sidecar 可执行文件（storyloom-sidecar.exe）
server/
  ├─ index.ts             Fastify 入口 + startServer()
  ├─ sidecar-entry.ts     Tauri sidecar 专用入口（子进程信号 + 日志重定向）
  ├─ db/                  drizzle + better-sqlite3 + migrations 调用
  ├─ plugins/             error-handler / database / validation
  └─ routes/<resource>/   每资源一个 plugin（多文件时用目录如 workspaces/）
src/
  ├─ components/<feature>/  按功能而非按类型组织（timeline/ outline/ workspace/ ...）
  ├─ components/_shared/    跨功能复用（注：当前以 ui/ 与 ui-tdesign/ 收容）
  ├─ lib/
  │   ├─ tauri-api.ts     Tauri API 统一封装（文件 / 端口 / 日志 / 对话框 / 更新）
  │   └─ ...              纯函数工具与领域模块
  ├─ services/api-hooks.ts  react-query hooks 单文件入口
  ├─ services/api.ts        fetch 封装 + 动态 base（适配 Tauri 端口发现）
  ├─ stores/                Zustand 切片
  └─ App.tsx                极简：仅挂 WorkspaceInitializer + AppShell + UpdateNotifier
scripts/                  一次性脚本，文件名带描述（如 generate-storyloom-icons.py）
shared/                   前后端共享类型
```

新增组件时优先放进对应 feature 目录；只有真正跨功能复用且粒度极小的（按钮、薄包装）才考虑 `ui/`。

## 5. 测试约定

- 框架：[vitest](https://vitest.dev/) + jsdom（见 `src/test/setup.ts`）。
- 文件命名：`*.test.ts(x)`，与被测源码并列（例：`WorkspaceCard.tsx` 旁边 `WorkspaceCard.test.tsx`）。
- 跑一次：

  ```bash
  npm run test
  ```

- 起 watch：

  ```bash
  npm run test:watch
  ```

- 类型检查（不出代码）：

  ```bash
  npm run typecheck
  ```

E2E / 视觉回归在 `scripts/` 下用 Python + Playwright 写，按需手动跑（如 `npm run test:e2e:chromium`）。

## 6. 提交约定

- [Conventional Commits](https://www.conventionalcommits.org/) + scope，例：

  ```
  feat(timeline): add gantt zoom slider
  fix(workspace): unblock create-workspace dialog catch handler
  chore(v3.0.0): cleanup repo + rewrite docs
  refactor(server): split workspaces routes into folder
  docs(architecture): refresh module map for v3.0
  test(workspace): add WorkspaceCard render snapshot
  ```

- 大改动写 `.trae/specs/<change-id>/` 三件套（design / requirements / tasks）。改动完成后 `.trae/specs/_archive/` 归档，避免活跃目录膨胀。
- 推送前最低门槛：

  ```bash
  npm run typecheck && npm run test
  ```

  发版前再加 `npm run build && npm run build:server && npm run build:sidecar`。

## 7. 常见坑

### (1) ESM 下相对 import 必须写 `.js` 后缀

仓库 `"type": "module"`，tsc 与 Vite 都按纯 ESM 解析；哪怕源文件是 `.ts`，相对路径里也要写 `.js`：

```ts
// ✅ 正确
import { startServer } from './server/index.js';
import { workspacesRoutes } from './routes/workspaces/index.js';

// ❌ 错误：运行时 ERR_MODULE_NOT_FOUND
import { startServer } from './server/index';
```

### (2) Rust 源码修改后 Tauri 开发模式自动重载

`npm run tauri:dev` 会自动监听 `src-tauri/src/` 下的 Rust 文件变化并重新编译。若编译失败，控制台会显示 cargo 错误信息。常见 Rust 编译错误：

- 类型不匹配：`tauri::command` 函数签名必须实现 `Serialize` / `Deserialize`
- 异步命令：使用 `async fn` 时需要 `tokio` runtime（已在 `Cargo.toml` 中配置）
- 路径错误：`tauri.conf.json` 中的路径是相对于 `src-tauri/` 目录的

### (3) better-sqlite3 无需 electron-rebuild

Tauri 时代不再需要 `electron-rebuild`。`better-sqlite3` 在 `npm install` 时自动编译为 Node.js 原生模块，由 sidecar 进程（Node.js）加载，与 Rust 主进程无关。

### (4) `data/` 路径在生产用 Tauri API

不要在 server 代码里硬编码 `./data/`。生产模式由 Rust 主进程注入：

```ts
// src-tauri/src/lib.rs
let data_dir = app_handle.path().app_local_data_dir()?.join("data");

// sidecar 启动时通过环境变量传入
env("DATA_DIR", &data_dir)
```

调试用户数据时，Windows 实际目录通常是 `%APPDATA%\com.storyloom.app\data\timeline.db`。

### (5) Sidecar 启动失败排查

Sidecar 启动失败时检查以下几点：

1. **`src-tauri/sidecars/storyloom-sidecar.exe` 是否存在**
   - 开发模式：`npm run dev:sidecar` 直接运行 `server/sidecar-entry.ts`
   - 生产模式：必须先 `npm run build:sidecar` 生成 sidecar 可执行文件

2. **环境变量是否正确注入**
   - `DATA_DIR`：数据目录绝对路径
   - `MIGRATIONS_DIR`：drizzle 迁移目录绝对路径
   - `NODE_ENV`：应为 `production`

3. **端口冲突**
   - sidecar 默认尝试 3001-3010 端口，若全部被占用会启动失败
   - 检查 `app.log` 或 sidecar stderr 输出中的端口错误信息

### (6) Tauri 开发窗口白屏

若 `npm run tauri:dev` 打开窗口后白屏：

1. 检查 Vite dev server 是否已启动（`http://localhost:5173` 可访问）
2. 检查 sidecar 是否已就绪（控制台应输出 `[dev-sidecar] Backend ready on port XXXX`）
3. 检查 Rust 控制台是否有 panic 信息
4. 尝试清除 `src-tauri/target/` 后重新编译

### (7) TDesign Dialog `onConfirm` 与 form `onSubmit` 不要重复绑同一动作

唯一入口走 `onConfirm`，form 用 `<button type="submit" hidden>` 兜底回车键，避免双触发被 `mutateAsync` inflight 静默吞掉：

```tsx
<Dialog onConfirm={handleSubmit} confirmBtn={{ disabled: !name, loading: applying }}>
  <form onSubmit={handleSubmit}>
    {/* fields */}
    <button type="submit" hidden />
  </form>
</Dialog>
```

`catch` 块里务必 `toast.error(message)` + `finally` 复位 loading 标志；否则用户感知"点了没反应"（v3.0 创建工作区 bug 真因）。
