# Storyloom · 絮织 — 长期开发路线图（v1.2+）

> 编写日期：2026-06-19（v1.1.4 改名 Storyloom + 视觉小说专属能力 + 试玩预览大方向）
> 编写者：Trae AI
> 目标读者：本仓库后续接手开发的 AI 与人类协作者
>
> **本次重大变更**：
> 1. **再次改名**：`AI Timeline Creator` → `Timeline Creator` → **`Storyloom · 絮织`**
>    - 双关：Story（故事）+ Loom（织机）。多轨道时间轴恰是经纬，故事是织出来的——这是项目最贴切的视觉与功能隐喻
>    - 中文名「絮织」：絮语 + 织造，柔软文学，绝不工具化
>    - "Timeline Creator"过于直白工具化，且容易和市面上一堆通用时间轴软件撞车
> 2. **聚焦视觉小说作者人群**——剧本（带分支） / 立绘 / 背景 CG / 对话 / 选项是视觉小说五要素，我们做"剧本与素材的预生产工作台"，最后**导出到主流引擎可消费的格式**
> 3. **未来大方向：内置试玩预览** — 用户在 Storyloom 中规划好一个章节后，可以直接"播放"该章节，像玩一个真正的视觉小说一样测试节奏，不必先导出再到引擎里跑
>
> **核心定位**：Windows 桌面端 + 单机优先 + 暖色纸感视觉的**视觉小说创作工作台**。**不做引擎本身，做引擎的上游**。**不做云、不做协作、不做生态、不做 AI 生成图**。

---

## 0. 调研结论（视觉小说作者真正需要什么）

主流视觉小说引擎现状：

| 引擎 | 类型 | 中文支持 | 我们是否能导出给它 |
|---|---|---|---|
| **Ren'Py** | Python，桌面/Web，免费开源 | 良好 | ✅ 能导出 .rpy 脚本 |
| **WebGAL** | Web 引擎，开源（中文社区） | 原生 | ✅ 能导出 .txt 脚本（首选目标） |
| **TyranoBuilder** | 商业，可视化拖拽 | 一般 | ✅ 能导出 .ks 脚本 |
| **Naninovel** | Unity 插件，商业 | 良好 | ✅ 能导出 .nani 脚本 |
| **Visual Novel Maker** | RPG Maker 团队出品 | 良好 | ⚠️ 私有格式，先不做 |
| **Godot** | 通用游戏引擎 | 良好 | ⚠️ 用户自己写脚本，不导出 |

**关键洞察**：
1. **作者的 80% 时间花在剧本 + 立绘排布上，不在引擎层** — 这正是 Storyloom 的位置
2. **国人作者首选 WebGAL** — Web 引擎，开源，中文社区活跃，无版权风险，**这是我们导出的首要目标**
3. **每个引擎脚本格式不同但语义相通**：人物说话、立绘切换、背景切换、选项分支、BGM/SFX——我们用一个抽象中间层（StoryloomScript IR），再翻译到各引擎
4. **试玩预览** — 作者最想要"这一段写完，能立刻看到主角立绘出场、配景切换的样子"，再决定是否调整节奏

## 1. 必要功能（要做）

### 1.1 v1.2 — 图片资产能力（视觉小说核心，已在上一版路线图列出）

详见上一版路线图保留内容（assets 表 + avatar/portrait/scene/map 四类、character_assets/event_assets 关联表、Lightbox 预览、图库浏览器、关系图节点头像、时间轴卡片场景图缩略、导出含图片）——P0 大模块，核心。

### 1.2 v1.2 — 视觉小说专属能力（新增，本次核心）

**为什么必要**：用户明确"为视觉小说专门设计"，未来可"进行视觉小说的测试"。当前我们的"事件/轨道/大纲"模型偏小说作家；要把它扩展到视觉小说编剧的工作模型。

#### 1.2.1 数据模型扩展

新增视觉小说专属对象：

| 对象 | 字段 | 说明 |
|---|---|---|
| **Scene（场景）** | id / name / background_asset_id / bgm / order | 一个视觉小说场景，对应一段时间内的同一背景 |
| **Beat（节拍）** | id / scene_id / kind (line/choice/jump/sfx/anim) / character_id / portrait_asset_id / text / metadata | 场景内最小叙事单元，相当于一行剧本 |
| **Choice（选项）** | id / beat_id / label / next_scene_id / condition | 分支选择 |
| **Flag（旗标）** | id / name / default_value | 用户在游戏中产生的状态变量（好感度、是否触发某事件等） |

事件（Event）→ Scene 集合：一个时间轴事件可映射成 1+ 个 Scene。**保持兼容**：用户可以选择"小说模式"（不启用 Scene/Beat），也可"视觉小说模式"。

#### 1.2.2 剧本编辑器（Beat-by-Beat Editor）

**新增 view**：`/script` — 视觉小说专属剧本视图。

```
┌──────────────────────────────────────────────────────────────┐
│ 场景：废弃教堂  [背景: church_dusk.png] [BGM: 雨声.ogg]       │
├──────────────────────────────────────────────────────────────┤
│ 立绘    │ 角色     │ 文本                                       │
│ ────    │ ────     │ ───                                        │
│ [雪麦]  │ 雪麦     │ "这里就是传闻里的地方？"                    │
│ [雪麦]  │ 雪麦     │ "夕阳从破洞的彩窗洒下来，像血。"           │
│         │ ——       │ [选择] ① 走近祭坛  ② 转身离开              │
│ [林]    │ 林（笑） │ "胆子倒是不小。"                            │
└──────────────────────────────────────────────────────────────┘
```

要做的：
- 每行 = 一个 Beat。可拖拽行重排 / 复制 / 删除
- 立绘列：从角色档案的 portrait 列表中选；可选「无立绘」「淡入」「淡出」
- 角色列：下拉选已建角色；可选"旁白"
- 文本列：富文本，支持 `{name}` 占位符（自动替换为角色名）
- 行类型切换（line / choice / jump / sfx / anim）
- 节拍快捷键：Enter 新建一行、Tab 切换列、Ctrl+D 复制行
- 行内字数统计（适合短句节奏）

#### 1.2.3 分支地图（Branch Map）

**新增 view**：`/branches` — 用图形化方式展示场景之间的跳转。

要做的：
- 节点 = Scene；连线 = Choice 或自动跳转
- 拖拽布局，自动算路径
- 支持 Flag 条件：连线上可标"if 好感度>=5"
- 死路 / 孤岛检测：高亮"无入口"或"无出口"的场景
- 导出：分支图 PNG / DOT 文件

#### 1.2.4 试玩预览（**未来大方向**）

**未来大方向，分两阶段实施**：

**v1.4 阶段 — 基础内置预览器**：
- 在剧本视图右侧加"播放"按钮
- 弹出独立窗口（720×1280 竖屏 或 1280×720 横屏，作者选）
- 用我们自己的轻量渲染器（pixi.js 或 canvas）按 Beat 顺序播放：背景图淡入 → 立绘登场 → 文本逐字显示 → 等待空格 → 下一句
- 支持选项分支跳转
- 不做 BGM 播放（v1.4 先简化）
- 用途：作者写完一段就能自己"过一遍剧本"，体感节奏

**v2.0 阶段 — 嵌入 WebGAL 引擎做完整试玩**：
- WebGAL 是开源 Web 引擎（Apache 2.0），可作为 webview 嵌入 Electron
- 实时把 Storyloom 的 IR 翻译成 WebGAL 脚本，写入 webview 内存目录
- webview 加载 WebGAL，进入完整体验（含 BGM/SFX/动画/存档）
- 这才是真正"视觉小说的测试" — 与最终成品几乎一致

**v3.0 阶段（可选）— 多引擎预览**：
- 针对 Ren'Py / TyranoBuilder 各做一个本地 sidecar 进程
- 用户偏好里选默认预览引擎
- 不做也不影响主线（WebGAL 已够好）

#### 1.2.5 导出到主流引擎

**v1.3 阶段 — WebGAL（首选）**：
- 把 IR 翻译成 WebGAL `.txt` 脚本
- 资产打包：把所有引用的 portrait/scene/bgm 复制到 `<output>/game/figure/`、`background/`、`bgm/`
- 输出整个 WebGAL 项目文件夹，作者拖到 WebGAL 编辑器即可继续编辑或 build

**v1.4 阶段 — Ren'Py**：
- 翻译成 `.rpy` 脚本
- Ren'Py 资产命名规范不同，做一层适配

**v1.5 阶段 — TyranoBuilder（按需）**：
- 翻译成 `.ks` 脚本
- 商业引擎，市场较小，按用户反馈再做

## 2. 不做的事（明确边界）

| 不做项 | 理由 |
|---|---|
| ❌ macOS / Linux / Web 在线版 / 移动端 | 聚焦 Windows |
| ❌ 云同步 / 多端协作 | 单机工具就是单机工具 |
| ❌ 插件市场 / 模板市场 / AI 模型市场 | 用户量不够 |
| ❌ AI 生成封面 / 立绘 / 场景图 | 用户更需要导入自己的图（插画师 / 付费素材 / 自绘） |
| ❌ **自研视觉小说引擎** | **WebGAL / Ren'Py 已经做得很好，我们做"上游"，不做"下游"** |
| ❌ 自研编程语言式剧本 DSL | 学习成本高，作者更喜欢"表格式"剧本 |
| ❌ 组织架构图 / 词条字典 / 随机名字生成器 | 受众太小 |
| ❌ Vitest 覆盖率硬指标 / Actions 自动 release | 自然推进 |

## 6. 路线图维护准则

> **少即是多**。每次想加东西时先问三个问题：
> 1. 用户真的会用吗？还是只是"看起来酷"？
> 2. 加上去后维护成本是多少？
> 3. 砍掉某项时是否真的会损失用户？
>
> 拒绝"为了完整而完整"。
>
> **新增第四问（v1.1.4）**：
> 4. 这件事是否符合 Storyloom 的"视觉小说编剧上游工作台"定位？还是飘向了引擎、生态、协作等远端？

## 7. 改名与迁移计划

| 阶段 | 项目名 | appId | 仓库 URL | userData 路径 |
|---|---|---|---|---|
| v1.0.x – v1.1.0 | AI Timeline Creator | `com.ai.timeline-creator` | `AI-Timeline-Creator` | `%APPDATA%\AI Timeline Creator\` |
| v1.1.3 | Timeline Creator（过渡） | 同上保留 | 同上保留 | `%APPDATA%\Timeline Creator\` |
| **v1.1.4+ 当前** | **Storyloom · 絮织** | 同上保留 | 同上保留 | **`%APPDATA%\Storyloom\`** |
| v2.0（未来） | Storyloom · 絮织 | `app.storyloom` | `Storyloom` | `%APPDATA%\Storyloom\` |

**已发布客户端如何升级到 Storyloom**：
- v1.1.0 用户启动后会通过 electron-updater 收到 v1.1.4 提示
- 接受升级后产品名变为 Storyloom，但 appId 不变 → Windows 视为同一应用，只是显示名变化
- userData 路径会从 `%APPDATA%\AI Timeline Creator\` 迁移到 `%APPDATA%\Storyloom\`，**首次启动时执行迁移脚本**（`electron/main.ts` 加一段：检测旧目录存在 → 复制到新目录 → 重命名旧目录为 `.bak`）—— 这条**必须实现**，否则用户数据会"丢失"

**v2.0 时**：
- 把 GitHub 仓库改名为 `Storyloom`（GitHub 自动 301 重定向，旧 URL 仍可访问）
- 把 appId 改为 `app.storyloom`，但**保留旧客户端兼容到 v2.x 末**
