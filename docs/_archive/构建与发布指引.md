# 构建与发布指引

> **⚠️ 必读：构建产物必须输出到仓库目录之外。**
>
> 在 Windows 下，仓库根目录会被 IDE / OneDrive / Windows Search Indexer / 防病毒 同时监听。一旦 `release/`、`win-unpacked/`、`app.asar` 出现在仓库内，下次 `npm run dist` 极易卡在
> `EBUSY: resource busy or locked, unlink ... app.asar`，最终导致打包失败、文件无法删除、Git 状态混乱。请始终将 electron-builder 输出到仓库之外（默认 `D:\AIKFCC\AI-Timeline-Creator-dist\<version>`）。

---

## 一、为什么必须隔离生产/开发环境

仓库根目录被多方同时监听，是 EBUSY 锁定的根本来源。已知锁定来源清单：

1. **IDE 文件监听** — VS Code / Trae 默认对项目根递归监听 `chokidar` watcher，`win-unpacked/` 与 `app.asar` 一旦写入立即被打开。
2. **Windows Search Indexer (`SearchIndexer.exe`)** — 默认对用户目录建索引，会在 asar 出现后短暂持有读句柄。
3. **OneDrive 同步** — 项目位于 OneDrive 子目录时，新文件会被云端代理打开上传，`unlink` 必报 EBUSY。
4. **防病毒实时扫描** — Defender / 第三方 AV 对新生成的 `.exe` / `.asar` 进行扫描，扫描期间持有独占句柄。
5. **残留 node / electron 进程** — 上一次 dev 或 dist 中断后留下的子进程仍持有 `app.asar` 与 `resources/` 目录。
6. **electron-builder 自身重入** — 同一目录连续两次打包，旧 `win-unpacked/` 未清理就被覆盖，触发自身锁定。

> 结论：**仓库根目录被多方监听，是 EBUSY 的源头。** 把构建产物搬到非监听目录，是从根本上解决 EBUSY 的唯一可靠方法。

---

## 二、推荐构建流程

1. 关闭所有 dev-server 与 IDE 文件监听（或暂时把项目目录从 IDE 移除）。
2. 设置环境变量，把构建产物指向仓库之外的隔离目录：

   ```powershell
   $env:ATC_DIST_DIR = "D:\AIKFCC\AI-Timeline-Creator-dist\4.1.0"
   ```

3. 执行打包脚本（脚本会先调用 `clean:release` 清理外部 dist 目录）：

   ```powershell
   npm run dist
   ```

4. 打包完成后，安装包路径为：

   ```text
   D:\AIKFCC\AI-Timeline-Creator-dist\4.1.0\AI Timeline Creator Setup 4.1.0.exe
   ```

## 三、诊断 EBUSY app.asar 的命令清单

打包失败时，先用以下命令定位是谁还在持有 `app.asar`：

```powershell
# 1. 列出所有 node.exe 残留进程（含命令行）
Get-CimInstance Win32_Process -Filter "Name='node.exe'" |
  Select-Object ProcessId, CommandLine | Format-List

# 2. Sysinternals handle.exe：直接查 app.asar 的持有者
handle.exe app.asar
# 或使用 NirSoft OpenedFilesView（GUI），过滤 app.asar / win-unpacked

# 3. 查找仍加载 win-unpacked 模块的进程
Get-Process | Where-Object { $_.Modules.FileName -like '*win-unpacked*' } |
  Select-Object Id, ProcessName, Path
```

定位到具体 PID 后，用 `Stop-Process -Id <PID> -Force` 终止；如果是 OneDrive / SearchIndexer / Defender 占用，请见第六节"推荐 IDE 设置"中的排除策略。

## 四、关键技巧：用 asar extract 解压绕过锁定

`app.asar` 本质上是带头部的拼接文件，被锁时 `unlink`/`del` 都会失败，但流式读取仍可用。利用这一点可以绕过句柄锁：

1. 使用 `@electron/asar` 命令行工具流式读出全部内容到临时目录；
2. 解压成功即可 `Remove-Item -Force` 原 asar，或将外层 `release/`、`win-unpacked/` 目录整体重命名后再删除。

完整命令示例：

```powershell
$asar = "D:\AIKFCC\AI-Timeline-Creator V4.0\release\dist\win-unpacked\resources\app.asar"
npx --yes @electron/asar extract $asar "$env:TEMP\app_extracted"
Remove-Item $asar -Force
```

技巧要点：

- 解压时是只读流式访问，不需要独占句柄，常常能在 OneDrive/Defender 仍持有的情况下成功；
- 解压后原 `app.asar` 即使无法直接删除，也可以先 `Rename-Item` 到 `app.asar.purge_<时间戳>`，下次打包前再清理；
- 整个外层目录（如 `release/dist/win-unpacked/`）也可整体改名，`electron-builder` 看到目录不存在即会重建。

## 五、清理脚本 scripts/clean-release.ps1 说明

执行 `npm run clean:release` 会调用 `scripts/clean-release.ps1`，流程如下：

1. **终止本项目的 node / electron 残留进程** — 按命令行包含项目根路径过滤，避免误杀其他项目；
2. **重命名目标 dist 目录** — 把 `$ATC_DIST_DIR`（默认 `D:\AIKFCC\AI-Timeline-Creator-dist\4.1.0`）和仓库内可能残留的 `release/` 改名为 `<dir>_purge_<时间戳>`，让 electron-builder 下次能直接重建；
3. **使用 .NET Directory.Delete 强制递归删除** — `[System.IO.Directory]::Delete($renamed, $true)` 比 `Remove-Item` 更稳定；
4. **失败回退到 asar extract 救援** — 若 `app.asar` 仍持有句柄，调用 `npx @electron/asar extract` 流式绕过，再删除外层目录。

调用方式：

```powershell
npm run clean:release
```

`npm run dist` 已自动前置该脚本，一般无需手动执行。

## 六、推荐 IDE 设置（可选）

为进一步降低 watcher 噪音、防止意外锁定，建议：

1. **VS Code / Trae** — 在 `.vscode/settings.json` 或工作区设置中追加：

   ```json
   {
     "files.watcherExclude": {
       "**/release/**": true,
       "**/win-unpacked/**": true,
       "**/*.asar": true,
       "**/dist-server/**": true,
       "**/electron-out/**": true
     }
   }
   ```

2. **OneDrive** — 在 OneDrive 设置中关闭对项目目录的同步（"选择文件夹"取消勾选），或将项目移出 OneDrive 根。
3. **Windows Search** — 在"索引选项 → 修改"里将项目根目录从索引列表中移除。
4. **Windows Defender** — 在"病毒和威胁防护 → 排除项"中加入仓库根与 `D:\AIKFCC\AI-Timeline-Creator-dist`。

## 七、常见问题

**Q1：按推荐流程仍然 EBUSY，怎么办？**
A：先检查项目所在路径是否在 OneDrive 内（资源管理器图标显示云朵 / 对勾）。若是，临时将 `ATC_DIST_DIR` 改到非 OneDrive 路径（脚本默认 `D:\AIKFCC\AI-Timeline-Creator-dist`，本身就在 OneDrive 之外），并把项目本身也搬出 OneDrive。

**Q2：`handle.exe` 找不到持有进程，但仍报 EBUSY？**
A：多半是 Windows Defender 实时扫描或 SearchIndexer 的瞬时句柄，重试 1～2 次通常即可通过；持续失败请将 `D:\AIKFCC\_dist` 加入 Defender 排除项。

**Q3：仓库内已有残留的 `release/` / `win-unpacked/` 删不掉？**
A：执行 `npm run clean:release`；若仍失败，按第四节"asar extract 解压绕过锁定"的命令手动救援，再 `Remove-Item -Recurse -Force release`。

**Q4：可以把 dist 目录设到网盘 / 网络驱动器吗？**
A：不建议。网盘代理同样会触发 EBUSY，且打包速度会显著下降。请使用本地物理盘的非同步目录（如 `D:\AIKFCC\AI-Timeline-Creator-dist`）。
